# Retrieval Ablation — retrieval-v3 (2026-07-17)

Dataset: **100 questions**, generated by `gemini/gemini-3.1-flash-lite`
from the ingested corpus (1241 docs /
7383 chunks; one question per document, gold label =
source chunk). Ranking depth k=10. Embeddings: `BAAI/bge-base-en-v1.5`
(dense) + `Qdrant/bm25` (sparse), reranker
`BAAI/bge-reranker-base` (see ADR-0003).

## Reading the numbers

- **Chunk-level is a lower bound.** The corpus contains near-duplicate content
  across documents (e.g. similar compose examples), so a retrieval that surfaces
  an equivalent-but-different chunk scores 0 at chunk level. Document-level
  credit corrects for the within-document half of that; cross-document
  duplicates still depress both.
- Latency is per query, sequential, CPU-only embeddings/rerank on the dev
  machine — comparative, not absolute.
- Raw per-case results: `retrieval-ablation-v3-raw.json`.

## What v3 is and what it settles

v3 = the 100 v2 questions **paraphrased by an LLM** (same meaning, different
incidental wording — mean vocabulary overlap with v2 only 0.31), gold labels
unchanged, same corpus. It exists to answer the question v2 raised: was
"sparse wins" real retrieval superiority, or vocabulary leakage from
one-question-per-chunk generation?

| chunk mrr@10 | v2 (verbatim) | v3 (paraphrased) | Δ |
|---|---|---|---|
| dense | 0.553 | 0.570 | +0.017 |
| sparse | **0.759** | 0.611 | **−0.149** |
| hybrid | 0.705 | **0.656** | −0.049 |
| hybrid+rerank | 0.745 | 0.645 | −0.100 |

**Verdict: it was leakage, and hybrid is the right default — now with
evidence.** Under paraphrase, sparse loses 20% of its MRR (its v2 win was
built on questions that reuse the gold chunk's exact tokens), dense is flat
(semantic vectors don't care about wording — its weakness is the corpus
scale, not paraphrase), and hybrid degrades 3× less than sparse because the
dense leg carries it when vocabulary shifts. Chat's default
(`SearchMode.HYBRID`, no rerank) stands on all three regimes: best on v1,
close second on v2, best on v3.

The honest bracketing: real user queries live somewhere between v2 (verbatim
error strings pasted into the box — sparse's home turf) and v3 (described
problems in the user's own words). Hybrid is the only mode that is
near-optimal at both ends; that is the decision rationale, not any single
table.

Rerank on the paraphrased set is quality-neutral-to-negative again
(0.656 → 0.645) at 10.6 s/query — the v2 "rerank helps" result does not
generalize across regimes, which reinforces keeping OPSVERSE_CHAT_RERANK
default-off.

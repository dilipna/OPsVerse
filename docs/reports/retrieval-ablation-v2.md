# Retrieval Ablation — retrieval-v2 (2026-07-14)

Dataset: **100 questions**, generated by `gemini/gemini-3.1-flash-lite`
from the ingested corpus (1241 docs /
7383 chunks; one question per document, gold label =
source chunk). Ranking depth k=10. Embeddings: `BAAI/bge-base-en-v1.5`
(dense) + `Qdrant/bm25` (sparse), reranker
`BAAI/bge-reranker-base` (see ADR-0003).

## Reading the numbers

- **Chunk-level is a lower bound.** The corpus contains near-duplicate content
  across documents (e.g. similar compose examples), so a retrieval that surfaces
  an equivalent-but-different chunk scores 0 at chunk level. Document-level
  credit corrects for the within-document half of that; cross-document
  duplicates still depress both.
- Latency is per query, sequential, CPU-only embeddings/rerank on the dev
  machine — comparative, not absolute.
- Raw per-case results: `retrieval-ablation-v2-raw.json`.

## v1 → v2: what changed and what we do about it

v1 (100 Qs, 144-doc/421-chunk original corpus) vs v2 (100 Qs, 1,241-doc/
7,383-chunk cleaned corpus — localized pages purged, English k8s concepts
re-ingested). Different question sets, so this compares *regimes*, not runs.

| chunk-level | v1 mrr@10 | v2 mrr@10 |
|---|---|---|
| dense | 0.636 | **0.553** |
| sparse | 0.605 | **0.759** |
| hybrid | **0.643** | 0.705 |
| hybrid+rerank | 0.612 | 0.745 |

**1. Sparse (BM25) wins at this scale — with a known bias caveat.** v2
questions are highly specific (exact flags, resource names, version pins),
and one-question-per-chunk generation tends to reuse the gold chunk's
vocabulary, which flatters lexical matching. Real users paraphrase more than
a generator staring at the chunk does. **Decision: chat stays on hybrid**
(robust to paraphrase; only 0.05 mrr behind sparse and equal hit@10) — but
sparse-only is now a measured, honest option if latency ever matters more,
at 8.5 ms/query it is 3× faster than hybrid. A paraphrased/adversarial
evalset is the right follow-up before switching defaults.

**2. Dense-only degraded the most as the corpus grew** (0.636 → 0.553):
more near-neighbor distractors at 768d. This retroactively justifies hybrid
over dense (ADR-0003) — the v1 gap was too small to prove it.

**3. The rerank verdict flips from "quality-negative" to "quality-positive,
latency-prohibitive".** On v1 rerank *hurt* (0.643 → 0.612); on v2 it helps
hybrid (0.705 → 0.745, hit@1 0.58 → 0.64) — cross-encoders earn their keep
once the candidate pool actually contains hard negatives. It still costs
~6.9 s/query on CPU vs a ≤300 ms rerank budget (docs/latency-budget.md), so
**OPSVERSE_CHAT_RERANK stays default-off**; the revisit trigger is a GPU
serving path or a distilled reranker, not more corpus.
